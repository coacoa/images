OpenWrt が突然ネットワークから消える原因と、再設定時の注意点（完全版）

OpenWrt（特に DSA モデル）では、設定変更やブリッジ構成の変更により、
突然ネットワークから消える現象が起きることがある。
ここでは、その原因と再設定時に気をつけるべきポイントをまとめる。

1. network restart による完全断線

/etc/init.d/network restart は、LAN/WAN/ブリッジをすべて DOWN → UP する。
その瞬間に以下が発生する：

  • LAN ポートが一時的にすべて DOWN
  • br-lan が再構築される
  • DHCP リースが消える
  • ARP テーブルが消える
  • 上流ルーターが OpenWrt を見失う

結果：SSH も WebGUI も消える。

2. DSA モデル特有の MAC アドレス変化

DSA では LAN と WAN が別デバイスで、MAC も別。
WAN を br-lan に追加したり、ブリッジを再構築すると、
br-lan の MAC が変わることがある。

上流ルーター（例：F660T）は古い MAC を保持したままになり、
通信不能になる。

3. DHCP リースの消失

restart により DHCP クライアントが再起動し、
以前の IP（例：192.168.1.250）が無効化される。
上流ルーターの DHCP リストから消える。

4. ARP テーブル破損

MAC が変わると、上流ルーターの ARP テーブルが壊れる。
結果として、OpenWrt に ping も通らず、SSH もできない。

5. WAN/LAN の誤接続による L2 ループ

WAN を LAN 化した状態で、上流ルーターへ複数本接続すると、
スイッチループが発生し、ネットワーク全体が停止する。

STP が OFF の場合は特に危険。

6. DSA のブリッジ再構築の重さ

DSA モデルはブリッジ再構築が重く、
20〜40秒ほど完全に通信不能になることがある。
その間に上流ルーターが OpenWrt を「死んだ」と判断する。

再設定時に絶対に気をつけるポイント

1. restart を使わず、reload を使う

SSH が切れない安全な反映方法：

/etc/init.d/network reload

2. WAN を LAN 化する前に STP を ON にする

uci set network.lan.stp='1'
uci commit network
/etc/init.d/network reload

ループ事故を防ぐために必須。

3. 上流ルーターへは必ず 1 本だけ接続

WAN と LAN の両方を上流に挿すと 100% ループ。

4. DHCP サーバーは必ず OFF

ブリッジ/AP モードでは OpenWrt 側の DHCP は不要。
二重 DHCP はネットワーク崩壊の原因。

5. DSA では swconfig は使えない

LAN と WAN は VLAN ではなく、
完全に別デバイスとして扱われる。

6. WAN を br-lan に追加する正しい方法

uci add_list network.lan.device='wan'
uci commit network
/etc/init.d/network reload

7. 上流ルーターの ARP/DHCP が壊れる前提で動く

何かおかしくなったら、上流ルーターを再起動すれば復旧することが多い。

まとめ

OpenWrt が突然消えるのは、設定ミスではなく、
DSA モデル特有の挙動・restart の仕様・MAC 変化・ARP 破損などが原因。
正しい手順で設定すれば安定して動作する。
