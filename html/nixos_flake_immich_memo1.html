1. 今回のトラブルと解決策のまとめ

現象 原因 解決策
file already closed /tmp内の一時的なログ書き込み失敗 -L ./pc.logでログ出力先を明示的に固定する
File exists (mkdir) ディレクトリ名の「ファイル」が既に存在 rm -rfで実体を確認し、mkdir -pで作り直す
invalid token (nix develop) コメント内のマルチバイト文字(日本語) Nixコード内の日本語コメントを排除しASCIIのみにする
errno: -3008 (ENOTFOUND) Redisの名前解決失敗 127.0.0.1を直接指定し、DNS解決をスキップさせる

2. 汎用的な構築指針 (Best Practices)

① ネットワーク接続は「IP指定」が最強

localhost や redis といったホスト名は、実行環境（Docker, Nix, Node.jsのバージョン等）によって名前解決に失敗することが多いです。 127.0.0.1 を直接指定するのが、最もエラーが起きにくい「安牌」の設定です。

② Nixの多行文字列内のエスケープ

Nixの '' ... '' 内で ''（空のクォート）を使いたい場合、直接書くとNixの終端と誤認されます。 '${"''"}' のように、Nixの変数展開を利用して「空の文字列」を注入するテクニックを使うと安全です。

③ process-compose の依存関係設定

  • 初期構築時は condition: process_started で「とりあえず全部動かす」方がエラーログが見えやすい。
  • 安定してきたら process_healthy に切り替えて、起動順序を制御する。

④ ディレクトリ・パーミッションの事前確認

process-compose 自体にログを書かせるのではなく、サービス専用のデータディレクトリ（例: ~/immich/logs）を用意し、そこを起点に構築するとデバッグが容易になります。

重要: 構築が詰まったら --tui=false で生のログを見ること。管理画面(TUI)で隠れてしまうエラーメッセージに答えがあります。
