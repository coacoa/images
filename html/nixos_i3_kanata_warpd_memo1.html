Kanata + i3 + warpd 構築メモ

目的

sxhkd を廃止し、 Kanata をキー入力専用、 i3 をコマンド実行専用 として統一する。

最初の構成

無変換 + H
      ↓
Kanata
      ↓
cmd warpd --hint
結果 
Could not connect to X server
原因
  • Kanata は systemd service として動作している
  • DISPLAY が無い
  • XAUTHORITY が無い
  • GUIアプリ(warpd)を起動できない

次の案

Kanata から F13 を送る。 
無変換 + H
      ↓
Kanata
      ↓
F13
      ↓
i3
      ↓
warpd --hint
Kanata 
warpd-hint f13
i3 
bindsym F13 exec --no-startup-id warpd --hint
結果

動かなかった。

調査

  • Kanata のログを見ると F13 は送信されていた。
  • i3 の bindsym も読み込まれていた。
  • しかし X11 側では F13 が正常に扱われていなかった。
つまり 
Kanata
   ↓
F13
   ↓
X11
   ↓
（ここで失われる）
環境依存で F13～F24 が使えないことがある。

最終解決

F13 をやめて Menu キーを使用。 Kanata 
warpd-hint menu
i3 
bindsym Menu exec --no-startup-id ${pkgs.warpd}/bin/warpd --hint
結果

正常動作。

最終構成

無変換 + H
      ↓
Kanata
      ↓
Menu (仮想キー)
      ↓
i3
      ↓
warpd --hint

なぜ Menu は動くのか？

  • Menu は X11 が標準でサポートする keysym。
  • 物理キーが無くても Kanata が仮想キーとして送信できる。
  • i3 が問題なく bindsym できる。
一方 F13～F24 は
  • XKB
  • uinput
  • キーボードレイアウト
によって認識されない場合がある。

得られた構成

  • sxhkd 不要
  • Kanata にショートカットを集約
  • GUI 起動は i3 が担当
  • X Server 問題を回避
  • 役割分担が明確

教訓

Kanata は「キー入力を変換するソフト」と考え、 GUI アプリの起動は Window Manager(i3) に任せる方が設計として自然で安定する。

仮想キーを経由する場合は、F13～F24 よりも Menu キーの方が環境依存が少なく、Linux/X11 では安定して利用できる。
