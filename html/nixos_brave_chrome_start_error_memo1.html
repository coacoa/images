NixOS / Home Manager ブラウザ起動トラブル解決メモ SOLVED

1. 根本的な原因（何が起きていたか）

原因は「同一モジュールの競合」と、それに伴う「プロファイルの多重ロック」のコンボでした。

① 設定の競合（Nix側の問題）

BraveもGoogle Chromeも、内部は同じChromiumベースのブラウザです。そのため、Home Managerの programs.chromium モジュールを両方のファイルで同時に定義（インポート）したことで設定の衝突が発生し、環境が不安定になりました。

② ゾンビプロセスとSingletonLock（Linux/ブラウザ側の問題）

不完全なビルドや起動の試行により、裏でブラウザのプロセスが死にきれずにゾンビ化（常駐）してしまいました。Chromium系のブラウザは、多重起動によるデータの破損を防ぐため、プロファイルディレクトリ内に SingletonLock というファイル（シンボリックリンク）を作成してロックをかけます。

プロセスが残っていた（あるいはロックファイルが消えずに残った）ため、新しくブラウザを起動しようとした際に、「既に別の場所でプロファイルが使われています」と判定され、安全のために起動がブロックされていました。

2. 解決したアプローチ

ステップ1：設定の分離（宣言の適正化）

programs.chromium の奪い合いを避けるため、構成を以下のようにスリム化しました。

  • Brave: programs.chromium を使って管理（拡張機能も宣言可能）
  • Chrome: home.packages = [ pkgs.google-chrome ]; で単体パッケージとして独立化

ステップ2：ロックの強制解除（プロセスの大掃除）

裏でプロファイルを掴んでいたゾンビプロセスを強制終了し、残留していたロックファイルを削除しました。

# 1. ゾンビプロセスをすべてキル
pkill -f brave
pkill -f chrome

# 2. 残ってしまったロックファイルを強制排除
rm -f ~/.config/BraveSoftware/Brave-Browser/SingletonLock
rm -f ~/.config/google-chrome/SingletonLock

3. 今後の教訓・メモ

  • Firefoxは独立した programs.firefox モジュールがあるため問題ないが、Chromium系（Brave, Chrome, Vivaldi等）を複数入れる時は programs.chromium の重複に注意する。
  • GUIでブラウザがウンともスンとも言わなくなった時は、まずターミナルから直接 brave や google-chrome を叩いて「エラーを吐いていないか」を確認するのが最速の解決ルート。
