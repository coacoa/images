Open Notebook on NixOS 構築メモ

ホスト: h370m（NixOS 26.05, NVIDIA GTX 1650 Super 4GB）
目的: NotebookLM 互換のセルフホスト環境を、ローカル LLM（Ollama）+ ローカル TTS（VOICEVOX）で完全ローカル運用する。

最終構成
Open Notebook（Web UI/API）+ SurrealDB（DB）+ Ollama（Chat/Embedding）+ VOICEVOX + 自作 FastAPI ラッパー（OpenAI互換TTS） すべて Docker コンテナ + systemd で NixOS 上に宣言的に管理。

1. Open Notebook 本体の導入

公式の docker-compose 構成（SurrealDB + open_notebook の2コンテナ）を NixOS モジュール化。virtualisation.oci-containers を使用。

configuration.opennotebook.nix
configuration.docker.nix   ← dockerHost モジュールとして分離

ポイント dockerHost モジュールを別ファイルに切り出し、services.dockerHost.enable = true; で他のサービスからも再利用できる形に。

問題1: services.openNotebook.enable = true; の呼び出し忘れでサービスが一切生成されなかった。
解決: モジュールの options 定義だけでは何も起動しない。configuration.h370m.nix 側で実際に enable する記述が必要。

2. コンテナ間ネットワーク問題

問題2: open-notebook コンテナから SurrealDB への接続が Name or service not known で失敗。
原因と解決: Docker のデフォルト bridge ネットワークではコンテナ名での名前解決ができない。専用ネットワーク open-notebook-net を systemd oneshot サービスで作成し、両コンテナの extraOptions に --network=open-notebook-net を追加して解決。

3. Ollama 導入（ローカル LLM）

GPU: GTX 1650 Super 4GB → pkgs.ollama-cuda を使用。 
services.ollama = {
  enable = true;
  package = pkgs.ollama-cuda;
  host = "0.0.0.0";
};
問題3: NixOS 26.05 で services.ollama.acceleration オプションが廃止されていてビルド失敗。
解決: package = pkgs.ollama-cuda に置き換え。
問題4: Open Notebook コンテナから Ollama (host.docker.internal:11434) に接続できず timeout。
原因が複数重なっていた:
  1. Ollama がデフォルトで 127.0.0.1 のみ listen → host = "0.0.0.0" で解消
  2. --add-host=host.docker.internal:host-gateway が SurrealDB コンテナにしか付いておらず、肝心の open-notebook コンテナに付いていなかった → 両方に追加
  3. NixOS firewall が 11434 番ポートを許可していなかった → allowedTCPPorts に追加

4. Podcast 機能（NotebookLM の Video/Audio Overview 相当）

調査の結果、Open Notebook には動画生成機能はなく、マルチスピーカー Podcast（音声）生成が NotebookLM の Deep Dive に相当する機能。

問題5: Podcast 生成が "Setup required"。Episode Profile に紐づくモデルが見つからない（NotFoundError: model:xxxx not found）。
解決: Settings → API Keys で Chat / Embedding / TTS モデルをそれぞれ登録し、Episode Profile / Speaker Profile の各モデル欄を選び直す。
問題6: Ollama は Chat と Embedding しか提供せず、TTS（音声合成）のモデルが一覧に出てこない。
方針決定: 完全ローカル運用にこだわるため、クラウド TTS（OpenAI 等）ではなく、既存の VOICEVOX 環境を活用する方向に。

5. VOICEVOX を OpenAI 互換 TTS サーバーに改修

Open Notebook の「OpenAI Compatible」プロバイダーは POST /v1/audio/speech エンドポイントを期待する。既存の VOICEVOX ラッパー（独自の /tts エンドポイント）を OpenAI 仕様に合わせて改修。

変更前変更後
POST /ttsPOST /v1/audio/speech
なしGET /v1/models 追加
speaker 固定（ID=3）voice パラメータ（alloy/echo/fable等）→ VOICEVOX speaker ID マッピング

構成は変わらず Podman 上の VOICEVOX Engine（ポート50021）+ FastAPI ラッパー（ポート8001、systemd管理）。

問題7: Open Notebook の Test Connection が timeout。
原因切り分けの手順:
  1. ExtraHosts がコンテナに反映されているか確認 → OK
  2. TTS API / VOICEVOX が起動しているか（systemctl status）→ OK
  3. ホストから直接 curl でTTS APIを叩けるか → OK
  4. コンテナ内から host.docker.internal:8001 に届くか → OK
最終的に NixOS firewall が 8001 番ポートを許可していなかったのが原因。allowedTCPPorts に追加して解決。
問題8: Podcast生成時に "nomic-embed-text:latest" does not support chat エラー。
原因: Embedding 専用モデルが、Episode Profile の Outline/Transcript Model（チャット用）の欄に誤って割り当てられていた。チャット用モデル（gemma:latest 等）に正しく差し替えて解決。

6. 最終確認

Podcasts タブから Generate Podcast を実行し、ステータスが Processing → Completed に変わることを確認。完了後、エピソードをクリックすると音声プレイヤーで再生・ダウンロードが可能。
